00047 /** A multi-dimensional domain over which to iterate. Used when
00048  * defining functions as reductions.
00049  *
00050  * A reduction is a function with a two-part definition. It has an
00051  * initial value, which looks much like a pure function, and an update
00052  * rule, which refers to some RDom. Evaluating such a function first
00053  * initializes it over the required domain (which is inferred based on
00054  * usage), and then runs update rule for all points in the RDom. For
00055  * example:
00056  *
00057  \code
00058  Func f;
00059  Var x;
00060  RDom r(0, 10);
00061  f(x) = x; // the initial value
00062  f(r) = f(r) * 2;
00063  Image<int> result = f.realize(10);
00064  \endcode
00065  *
00066  * This function creates a single-dimensional buffer of size 10, in
00067  * which element x contains the value x*2. Internally, first the
00068  * initialization rule fills in x at every site, and then the update
00069  * rule doubles every site.
00070  *
00071  * One use of reductions is to build a function recursively (pure
00072  * functions in halide cannot be recursive). For example, this
00073  * function fills in an array with the first 20 fibonacci numbers:
00074  *
00075  \code
00076  Func f;
00077  Var x;
00078  RDom r(2, 18);
00079  f(x) = 1;
00080  f(r) = f(r-1) + f(r-2);
00081  \endcode
00082  *
00083  * Another use of reductions is to perform scattering operations, as
00084  * unlike a pure function declaration, the left-hand-side of a
00085  * reduction definition may contain general expressions:
00086  *
00087  \code
00088  ImageParam input(UInt(8), 2);
00089  Func histogram;
00090  Var x;
00091  RDom r(input); // Iterate over all pixels in the input
00092  histogram(x) = 0;
00093  histogram(input(r.x, r.y)) = histogram(input(r.x, r.y)) + 1;
00094  \endcode
00095  *
00096  * A reduction definition may also be multi-dimensional. This example
00097  * computes a summed-area table by first summing horizontally and then
00098  * vertically:
00099  *
00100  \code
00101  ImageParam input(Float(32), 2);
00102  Func sum_x, sum_y;
00103  Var x, y;
00104  RDom r(input);
00105  sum_x(x, y)     = input(x, y);
00106  sum_x(r.x, r.y) = sum_x(r.x, r.y) + sum_x(r.x-1, r.y);
00107  sum_y(x, y)     = sum_x(x, y);
00108  sum_y(r.x, r.y) = sum_y(r.x, r.y) + sum_y(r.x, r.y-1);
00109  \endcode
00110  *
00111  * You can also mix pure dimensions with reduction variables. In the
00112  * previous example, note that there's no need for the y coordinate in
00113  * sum_x to be traversed serially. The sum within each row is entirely
00114  * independent. The rows could be computed in parallel, or in a
00115  * different order, without changing the meaning. Therefore, we can
00116  * instead write this definition as follows:
00117  *
00118  \code
00119  ImageParam input(Float(32), 2);
00120  Func sum_x, sum_y;
00121  Var x, y;
00122  RDom r(input);
00123  sum_x(x, y)   = input(x, y);
00124  sum_x(r.x, y) = sum_x(r.x, y) + sum_x(r.x-1, y);
00125  sum_y(x, y)   = sum_x(x, y);
00126  sum_y(x, r.y) = sum_y(x, r.y) + sum_y(x, r.y-1);
00127  \endcode
00128  *
00129  * This lets us schedule it more flexibly. You can now parallelize the
00130  * update step of sum_x over y by calling:
00131  \code
00132  sum_x.update().parallel(y).
00133  \endcode
00134  *
00135  * Note that calling sum_x.parallel(y) only parallelizes the
00136  * initialization step, and not the update step! Scheduling the update
00137  * step of a reduction must be done using the handle returned by
00138  * \ref Func::update(). This code parallelizes both the initialization
00139  * step and the update step:
00140  *
00141  \code
00142  sum_x.parallel(y);
00143  sum_x.update().parallel(y);
00144  \endcode
00145  *
00146  * When you mix reduction variables and pure dimensions, the reduction
00147  * domain is traversed outermost. That is, for each point in the
00148  * reduction domain, the inferred pure domain is traversed in its
00149  * entirety. For the above example, this means that sum_x walks down
00150  * the columns, and sum_y walks along the rows. This may not be
00151  * cache-coherent, but you may not reorder these dimensions using the
00152  * schedule, as it risks changing the meaning of your function (even
00153  * though it doesn't in this case). The solution lies in clever
00154  * scheduling. If we say:
00155  *
00156  \code
00157  sum_x.compute_at(sum_y, r.y);
00158  \endcode
00159  *
00160  * Then the sum in x is computed only as necessary for each scanline
00161  * of the sum in y. This not only results in sum_x walking along the
00162  * rows, it also improves the locality of the entire pipeline.
00163  */
