15 /** \file
16  *
17  * This file exports all routines which can be replaced by an
18  * application hosting code generated by %Halide. These are used when
19  * doing Ahead Of Time (AOT) compilation and must be supplied to the
20  * linker to override a routine. I.e., just define your own version of
21  * any of these functions with extern "C" linkage, and it should
22  * replace the default one.
23  *
24  * When doing Just In Time (JIT) compilation methods on the Func being
25  * compiled must be called instead. The corresponding methods are
26  * documented below.
27  *
28  * All of these functions take a "void *user_context" parameter as their
29  * first argument; if the Halide kernel that calls back to any of these
30  * functions has been compiled with the UserContext feature set on its Target,
31  * then the value of that pointer passed from the code that calls the
32  * Halide kernel is piped through to the function.
33  *
34  * Some of these are also useful to call when using the default
35  * implementation. E.g. halide_shutdown_thread_pool.
36  *
37  * Note that some linker setups may not respect the override you
38  * provide. E.g. if the override is in a shared library and the halide
39  * object files are linked directly into the output, the builtin
40  * versions of the runtime functions will be called. See your linker
41  * documentation for more details. On Linux, LD_DYNAMIC_WEAK=1 may
42  * help.
43  *
44  */

151 /** Called when Funcs are marked as trace_load, trace_store, or
152  * trace_realization. See Func::set_custom_trace. The default
153  * implementation either prints events via halide_printf, or if
154  * HL_TRACE_FILE is defined, dumps the trace to that file in a
155  * yet-to-be-documented binary format (see src/runtime/tracing.cpp to
156  * reverse engineer the format). If the trace is going to be large,
157  * you may want to make the file a named pipe, and then read from that
158  * pipe into gzip.
159  *
160  * halide_trace returns a unique ID which will be passed to future
161  * events that "belong" to the earlier event as the parent id. The
162  * ownership hierarchy looks like:
163  *
164  * begin_realization
165  * produce
166  * store
167  * update
168  * load/store
169  * consume
170  * load
171  * end_consume
172  * end_realization
173  *
174  * Threading means that ownership cannot be inferred from the ordering
175  * of events. There can be many active realizations of a given
176  * function, or many active productions for a single
177  * realization. Within a single production, the ordering of events is
178  * meaningful.
179  */
